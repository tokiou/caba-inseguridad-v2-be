# Auth Specification

## Purpose

Expose `/api/v1/auth/*`: email + password accounts with short-lived JWT access tokens and opaque,
rotating refresh tokens (stored hashed). Provide middleware that gates protected endpoints — today
`GET /api/v1/routes/safe` — on a valid, active user. Lives in `internal/auth/`
(`handler → service → repository`), with relational CRUD generated by sqlc (`internal/auth/db`).

## Requirements

### Requirement: User registration

The system SHALL expose `POST /api/v1/auth/register` accepting `{email, password}` and creating a
user with a bcrypt-hashed password. Email SHALL be unique. The response SHALL return the new user's
`id` and `email` and SHALL NOT include the password or password hash.

#### Scenario: New account is created

- GIVEN no user exists with email `test@example.com`
- WHEN a client POSTs `{ "email": "test@example.com", "password": "password123" }` to `/api/v1/auth/register`
- THEN the response is `201` with `{ "id": "<uuid>", "email": "test@example.com" }`
- AND the stored `password_hash` is a bcrypt hash, never the plaintext

#### Scenario: Duplicate email is rejected

- GIVEN a user already exists with email `test@example.com`
- WHEN a client registers the same email
- THEN the response is `409` with error code `email_taken`

#### Scenario: Invalid input is rejected

- WHEN a client registers with a missing/blank email or a password shorter than the minimum length
- THEN the response is `400` with error code `invalid_request`
- AND no user is created

### Requirement: Login issues an access token and a refresh cookie

The system SHALL expose `POST /api/v1/auth/login` accepting `{email, password}`. On valid credentials
for an active user it SHALL return a 15-minute JWT access token in the body
(`{access_token, token_type: "bearer", expires_in: 900}`) and SHALL set an HttpOnly refresh-token
cookie whose opaque value is stored only as a SHA-256 hash in a new `refresh_sessions` row. Every
login attempt SHALL be recorded in `login_attempts` (success and failure). Credential errors SHALL be
indistinguishable whether the email is unknown or the password is wrong.

#### Scenario: Successful login

- GIVEN an active user with valid credentials
- WHEN they POST correct `{email, password}` to `/api/v1/auth/login`
- THEN the response is `200` with `access_token`, `token_type: "bearer"`, `expires_in: 900`
- AND a `Set-Cookie` for the refresh token with `HttpOnly`, `SameSite`, `Path=/api/v1/auth`
- AND a `refresh_sessions` row exists holding the SHA-256 hash of the cookie value, not the value
- AND a successful `login_attempts` row is recorded

#### Scenario: Wrong password or unknown email

- WHEN a client logs in with a wrong password or an email that does not exist
- THEN the response is `401` with error code `invalid_credentials` (identical in both cases)
- AND a failed `login_attempts` row is recorded

#### Scenario: Inactive account cannot log in

- GIVEN a user with `is_active = false`
- WHEN they submit correct credentials
- THEN the response is `403` with error code `account_inactive`

### Requirement: Access token is a short-lived JWT

The access token SHALL be a JWT signed HS256 with `JWT_SECRET`, carrying claims `sub` (user id), `typ`
`"access"`, `iat`, `exp` (15 minutes after issue), and `jti`. Access tokens SHALL NOT be stored
server-side.

#### Scenario: Access token validates and expires

- WHEN an access token is minted
- THEN it verifies against `JWT_SECRET`, has `typ = "access"`, and `exp = iat + 15min`
- AND after `exp` it is rejected as expired

### Requirement: Refresh rotates the session

The system SHALL expose `POST /api/v1/auth/refresh` with no body, reading the refresh token from the
cookie. It SHALL look the session up by SHA-256 hash, reject the request `401` if the session is
missing, revoked, or expired, and otherwise SHALL rotate: mark the current session
`revoked_at = now()`, create a new session, link the old to the new via `replaced_by`, set a new
cookie, and return a fresh access token.

#### Scenario: Valid refresh rotates and returns a new access token

- GIVEN a valid, unexpired, unrevoked refresh cookie
- WHEN the client POSTs `/api/v1/auth/refresh`
- THEN the response is `200` with a new `access_token`
- AND a new `Set-Cookie` refresh token is issued
- AND the previous session is `revoked_at`-stamped with `replaced_by` pointing at the new session

#### Scenario: Revoked or expired refresh token is rejected

- GIVEN a refresh token whose session is revoked or past `expires_at`
- WHEN the client POSTs `/api/v1/auth/refresh`
- THEN the response is `401` with error code `invalid_refresh`
- AND no new session is created

### Requirement: Logout revokes the session

The system SHALL expose `POST /api/v1/auth/logout`, which reads the refresh cookie, revokes the
associated session (`revoked_at = now()`), clears the cookie, and returns `{ "message": "logged out" }`.
After logout the refresh token SHALL no longer be usable.

#### Scenario: Logout invalidates the refresh token

- GIVEN a logged-in client with a valid refresh cookie
- WHEN they POST `/api/v1/auth/logout`
- THEN the response is `200` with `{ "message": "logged out" }` and the cookie is cleared
- AND a subsequent `/api/v1/auth/refresh` with the old token returns `401`

### Requirement: Authenticated user endpoint

The system SHALL expose `GET /api/v1/auth/me`, protected by the auth middleware, returning the
authenticated user's `id` and `email`.

#### Scenario: Me returns the current user

- GIVEN a valid access token for an active user
- WHEN the client GETs `/api/v1/auth/me` with `Authorization: Bearer <token>`
- THEN the response is `200` with `{ "id", "email" }`

### Requirement: Access-token middleware

The system SHALL provide auth middleware that reads `Authorization: Bearer <jwt>`, validates the JWT
signature and `exp`, requires `typ = "access"`, loads the user by `sub`, rejects inactive or unknown
users, and injects the current user into the request context for downstream handlers. Missing or
invalid tokens SHALL yield `401` with error code `unauthorized`.

#### Scenario: Request without a token is rejected

- WHEN a protected endpoint is called with no `Authorization` header
- THEN the response is `401` with error code `unauthorized`

#### Scenario: Invalid or expired token is rejected

- WHEN a protected endpoint is called with a malformed, wrong-signature, expired, or non-`access`
  (e.g. `typ != "access"`) token
- THEN the response is `401` with error code `unauthorized`

#### Scenario: Valid token injects the user

- GIVEN a valid access token for an active user
- WHEN a protected endpoint is called
- THEN the middleware passes the request through with the current user available in context

### Requirement: Auth events are auditable

The system SHALL record auth events (registration, login success/failure, refresh, logout) such that
`login_attempts` captures every login outcome and `audit_logs` captures user-attributable actions
with optional `metadata`. Recording failures SHALL NOT break the user-facing request.

#### Scenario: Login outcomes are recorded

- WHEN any `/api/v1/auth/login` request completes
- THEN a `login_attempts` row records the email, outcome (`success`), and a `reason` on failure

### Requirement: Layered architecture

The capability SHALL live in `internal/auth/` following
`handler → service → repository interface → PostgresRepository (pgx)`, with routes registered in
`internal/app/routes.go`. Secrets (`JWT_SECRET`, `Authorization`, password, refresh token) SHALL never
be logged.

#### Scenario: Layers are not skipped

- WHEN an `/api/v1/auth/*` request is served
- THEN the handler parses HTTP and maps errors, the service holds the auth logic, and the repository
  performs all data access
